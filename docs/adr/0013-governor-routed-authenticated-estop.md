# ADR-0013: Software emergency-stop is an authenticated REQUEST to the governor, never a console→actuator command

| Field | Value |\n|---|---|\n| Status | **Proposed (design note — precedes code)** — for owner sign-off; ratified on merge. |\n| Date | 2026-06-21 |\n| Deciders | Project / safety-case owner |\n| Issues | #412 (this), #314 (operator identity), #410 / #405 (MRC / decel-to-stop), ADR-0006 (the QM↔safety boundary), #411 (console safety-theater fix) |\n| Code (when built) | `src/verifier.rs` / `src/bin/kirra_verifier_service.rs` (governor-request endpoint, mirrors the clearance loop), the audit chain, the MRC vocabulary (`TRAJECTORY_MRC_FALLBACK`) |\n\n## Principle\n\nAn emergency stop must **NOT** be a console→actuator command. The console is QM-domain and\nholds **no** actuator authority — that boundary is the core of KIRRA's ASIL-D safety case\n(ADR-0006). A software E-stop enters as an **authenticated REQUEST to the fail-closed\ngovernor**, which then commands the MRC under **its own** authority. The console asks; the\ngovernor acts. The console never touches the actuator.\n\nThis is the clearance loop, **inverted** — same channel, different verb:\n\n- **Clearance:** authenticated operator requests RELEASE → governor validates → governor acts\n  (resume motion).\n- **E-stop:** authenticated operator requests STOP → governor commands MRC → governor acts\n  (controlled stop).\n\nBoth route operator intent through the governor as the **sole authority**, over the\nauthenticated-request channel that already exists (operator identity, Ed25519-signed path,\nverify-then-consume, chain logging).\n\n## Decision\n\nAdopt a **governor-routed authenticated emergency-stop REQUEST**, bound by these hard\nconstraints (which **are** the safety argument):\n\n1. **REQUEST, not command.** The operator's stop is an input the governor *judges*, not a\n   command that reaches the actuator. The governor executes the MRC\n   (`TRAJECTORY_MRC_FALLBACK` / decel-to-stop·hold) under its own authority.\n2. **Authenticated + non-repudiable.** Same Ed25519 operator-identity path as clearance grants\n   — signed, verify-then-consume; the operator who requested the stop is provable in the chain.\n3. **Chain-logged.** A distinct audit event pair (`OperatorStopRequested` →\n   `GovernorMRCCommanded`) written to the tamper-evident chain — reconstructable after the\n   fact: who requested the stop, when, and what the governor did.\n4. **Supplementary to a PHYSICAL E-stop.** The certifiable emergency stop is a hardwired\n   circuit independent of compute (mushroom button → cut power / engage brakes). This software\n   request is a supervision/convenience layer **on top of** the physical E-stop, **never** the\n   primary safety mechanism. Stated explicitly so an assessor sees we understand the\n   certifiable stop is hardware.\n5. **Fail-closed.** If the request channel is down, the physical E-stop and the governor's\n   autonomous detection (SG1–SG6) still hold. The software request **adds** capability; its\n   absence removes nothing.\n\n## Rejected alternative (the anti-pattern)\n\nA **direct console→actuator stop command.** It would give the QM-domain console actuator\nauthority, punch a hole in the hypervisor boundary (ADR-0006), and force the safety case to\naccount for a compromised / laggy browser, session, or network commanding the vehicle. This is\nthe easy version and it is **forbidden** — it destroys the property that makes KIRRA\ncertifiable. (The console correctly has **no** E-stop button today; the dead safety-theater\nbutton was removed and only the E-stop *readiness indicator* remains — #411.)\n\n## Console framing (when built)\n\nThe console gains an authenticated **\"Request Emergency Stop\"** action that goes through the\ngovernor-request channel — framed in the UI as a **REQUEST** (\"Request Stop · routed to\ngovernor\"), never as a direct command, and gated behind operator authentication exactly like\nclearance.\n\n## Reuse (not a new mechanism)\n\n- **Mirrors:** the clearance loop (`verifier.rs`, `kirra_verifier_service.rs`).\n- **Reuses:** operator identity (#314), the Ed25519 signed path, verify-then-consume, the\n  audit chain, the MRC vocabulary (`TRAJECTORY_MRC_FALLBACK`).\n- **Posture:** an operator stop request drives the vehicle to a governor-commanded safe state;\n  relate to `FleetPosture`. Ties to the teleop-supervision cluster — operator intent is always\n  *governed*, never a bypass.\n\n## Phase scope / sequencing\n\nThis **design note first** (this ADR). Then implementation (laptop; end-to-end\n**hardware-gated** — needs a running governor + actuator path):\n\n1. the governor-request endpoint (mirror the clearance endpoint),\n2. the signed-request path (reuse the operator-identity verify-then-consume),\n3. the chain events (`OperatorStopRequested` → `GovernorMRCCommanded`),\n4. the console \"Request Emergency Stop\" action.\n\nCross-ref: #314, the clearance-loop PRs, ADR-0006 (the boundary), #411 (console\nsafety-theater fix).\n\n## Status\n\n**Proposed — for owner sign-off** (merge ratifies, as with ADR-0011 / ADR-0012). This records\nthe **request-not-command** architecture *before* code so the implementation cannot drift into\nthe console→actuator anti-pattern.\n
