# Reflash safety — how a wipe stays non-destructive

The point of this file: **getting the R2 image means reflashing the robot**,
and a reflash must not cost us the working KIRRA state that was hard-won on
the current image. This is the backup story that makes that safe.

## What lives where

| Category | Examples | Survives a reflash? |
|---|---|---|
| **In git (safe)** | The consumer/FFI/scripts (`robot/`), the install tooling (`robot/install/`), `env.template` with every validated value, the Rust sources for the cdylib + mint binary, the lidar truth (`installer/platform_map.toml:31-37`), the bringup + findings docs | ✅ always |
| **Robot-local, CAPTURED by Layer A** | The working ydlidar yaml, the vendor udev rules (`/dev/myserial`, `/dev/ydlidar`), device-symlink truth, version stamps | ✅ once `capture_from_robot.sh` output is committed |
| **Robot-local, REGENERATED by Layer A** | `/opt/kirra/*` (built `.so`, mint binary, installed scripts), `/etc/kirra/robot.env`, the staged systemd unit | ✅ re-created by `install_kirra.sh` |
| **Vendor-image-only** | ROS 2 Humble, `Rosmaster_Lib`, `ydlidar_ros2_driver`, the vendor udev rules as originally shipped, the board firmware behavior (car-type semantics) | ❌ comes from the flashed image — this is exactly why the captured copies matter |
| **Secrets** | `/etc/kirra/kirra.env` (verifier admin token / reset key — `deploy/systemd/install.sh:61-79`), any real governor signing key | ❌ **never committed**; regenerate (secrets) or re-enroll (keys) after a reflash |

## Before you reflash (checklist)

1. **Capture**: `robot/install/capture_from_robot.sh` on the working robot;
   review and **commit** `robot/install/captured/<hostname>/`.
2. **Push** everything: the repo clone on the robot may be the only copy of
   uncommitted work. `git status` must be clean, `git push` done.
3. Note anything odd in the capture output (missing yaml/udev warnings) and
   resolve it manually before wiping — after the wipe is too late.
4. Secrets: nothing to back up by design (regenerate after), but if a REAL
   governor key was provisioned (`file:` source), its seed lives in your key
   ceremony records, not on the robot alone — verify that before wiping.

## After the reflash (restore procedure)

1. Flash the new vendor base image (e.g. the R2 image, once obtained).
2. First boot: confirm the vendor pieces (`ROS 2`, `Rosmaster_Lib`,
   `ydlidar_ros2_driver`) and the device symlinks. If the new image's udev
   rules differ, restore the captured ones from
   `robot/install/captured/<hostname>/udev/`.
3. Clone the repo, then run **Layer A**:
   `robot/install/install_kirra.sh --dev-key` (bench) — this rebuilds the
   `.so` + mint binary and re-renders `/etc/kirra/robot.env`.
4. If the new image's ydlidar params differ from the captured working ones,
   restore from `robot/install/captured/<hostname>/ydlidar/`.
5. Re-run the verification (README §3): consumer OWNS line, `/scan` at
   ~10 Hz, then `first_run_elevated.sh` — 🔴 wheels elevated, always, on a
   fresh base.
6. **On the R2 image specifically**: the car-type register will read 5, so
   the Layer-A env (`KIRRA_EXPECTED_CAR_TYPE=1`) will make the consumer
   REFUSE to start — that is the guard working, not a bug. Layer B
   (`PLATFORM_R2_PENDING.md`) is the deliberate, validation-gated path to
   flipping that value; do not hand-edit it to 5 without running the Layer-B
   validation steps.
